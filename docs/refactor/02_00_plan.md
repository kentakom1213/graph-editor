# Phase 2 Refactor Plan: モデル分離と状態分割

対象: docs/refactor/00_01_checklist.md の Phase 2 チェックリストを詳細化する。

## ゴール
- `GraphEditorApp` に集中している状態を分割し、責務を明確化する。
- ドメインモデル (`GraphModel`) と UI 状態 (`GraphViewState`) を分離する。
- `Vertex` / `Edge` から UI 依存のフィールドを移し、純粋なグラフモデルにする。
- `GraphEditorApp` をオーケストレーターとして薄くする。

## 現状把握 (対象ファイル)
- 状態集中:
  - `src/app.rs` の `GraphEditorApp` が UI/ドメイン/エクスポート/入力状態を保持
- ドメインと UI の混在:
  - `src/graph/structures.rs` の `Vertex` / `Edge` が `is_pressed`, `is_selected`, `color`, `z_index`, `drag` など UI 依存を保持
- 描画/操作の分離は Phase 1 で進んだが、状態モデルは未分離

## 方針 (決めるべきこと)
- 状態分割の境界を明確化する。
  - `AppState`: ドメイン操作とグラフ自体
  - `UiState`: UI 入力/表示/パネル状態
  - `ExportState`: エクスポート処理の状態
- `GraphModel` と `GraphViewState` を分ける。
  - `GraphModel`: 頂点/辺の純粋データ
  - `GraphViewState`: 選択/ホバー/色/並び順/ドラッグ中の一時状態
- 既存 API を段階的に移行し、動作を壊さないように小さな単位で進める。

## 実施計画

### 1. 状態分割方針の確定
- `GraphEditorApp` のフィールドを分類して移動先を決める。
  - `graph` / `is_animated` / `next_z_index` / `edit_mode` / `selected_color` / `zero_indexed` / `show_number` / `cursor_hover` / `config` / `input_text` / `panel_tab` / `export_format` / `export_in_progress` / `export_request` / `error_message` / `confirm_clear_all`
- 新しい struct 設計案を作成する。
  - 例: `AppState { graph_model, view_state, edit_state }`
  - 例: `UiState { panel_tab, cursor_hover, input_text, error_message, confirm_clear_all }`
  - 例: `ExportState { format, in_progress, request }`
- 保存/復元 (`UiState` のシリアライズ) をどの層で扱うか決める。

### 2. `GraphModel` と `GraphViewState` の設計
- `GraphModel` に残す情報
  - 頂点 ID / 位置 / エッジ (from/to) / グラフ属性 (有向/無向)
- `GraphViewState` に移す情報
  - 頂点: `is_pressed`, `is_selected`, `z_index`, `drag`, `color`
  - 辺: `is_pressed`, `color`
- `GraphSnapshot` など描画用ビューがどちらに依存するか整理
  - 描画時は `GraphModel` + `GraphViewState` から `RenderSnapshot` を構築

### 3. UI 依存フィールドの移動
- `Vertex` / `Edge` から UI 依存フィールドを削除し、`GraphViewState` に移動
- 参照方法の再設計
  - `VertexId` (暫定: `usize`) に対応する `HashMap<VertexId, VertexView>` 形式
  - 辺の view 状態も `(from, to)` か `EdgeId` に紐付ける
- `GraphModel` / `GraphViewState` 同期の責務を定義
  - 例: 頂点削除時に view 状態も削除

### 4. `GraphEditorApp` の薄型化
- `GraphEditorApp` を「状態束ね + 画面遷移オーケストレーター」に変更
- 具体的には以下を別 struct に移管
  - 入力状態 → `UiState`
  - エクスポート状態 → `ExportState`
  - グラフ操作状態 → `GraphState` or `AppState`
- `components/*` が直接 `GraphEditorApp` に触れている箇所を順次置き換え

### 5. 移行手順と安全策
- 小さな移行単位で動作確認できる順序にする
  1. 新 struct を追加 (空実装) し、`GraphEditorApp` から参照
  2. UI 状態 (`panel_tab` / `cursor_hover` / `input_text`) を移動
  3. Export 状態を `ExportState` へ移動
  4. `GraphViewState` を追加し、`Vertex` / `Edge` の UI フィールドを段階的に移す
  5. `GraphSnapshot` 生成を `GraphModel + GraphViewState` で構築
- 変更のたびに `cargo check` を通す

## 影響範囲と注意点
- ID 再採番が UI 状態の参照と衝突しやすいので、view state の更新順に注意。
- `EditMode::AddEdge` などは UI 状態に依存するため、設計に合わせて整理が必要。
- `export.rs` / `graph_io.rs` は `GraphModel` を参照するように整理する。

## 完了条件
- `GraphEditorApp` が状態の直接保持を減らし、役割が整理されている。
- `Vertex` / `Edge` から UI 依存フィールドが移動している。
- 描画/操作が `GraphModel` + `GraphViewState` を利用する構造になっている。
- `cargo check` が通る。

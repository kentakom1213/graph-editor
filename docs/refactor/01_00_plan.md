# Phase 1 Refactor Plan: 描画とデータ更新の分離

対象: docs/refactor/01_plan.md の Phase 1 チェックリストを詳細化する。

## ゴール
- 描画ループから破壊的更新を排除し、描画は不変スナップショット参照に限定する。
- 削除処理などのメンテナンスを「操作フェーズ」で明示的に実行できるようにする。
- 既存挙動の差分を最小テスト/手動手順で確認できる状態にする。

## 現状把握 (対象ファイル)
- 描画中に `Graph::restore_graph()` が呼ばれている。
  - `src/components/central_panel.rs` の `draw_edges()` / `draw_vertices()`
- `Graph::restore_graph()` は「削除済み要素の除去 + 頂点ID振り直し + 辺ID補正」を行う。
  - `src/graph/structures.rs`
- `Graph::encode()` 内でも `restore_graph()` を呼び、エクスポート時にも破壊更新が走る。
  - `src/graph/structures.rs`

## 方針 (決めるべきこと)
- メンテナンスのタイミングを「入力/操作フェーズの終端」に寄せる。
  - 例: `GraphEditorApp::update()` の描画前に 1 回だけ適用。
- 描画は「削除フラグ付きの現行データ」ではなく「メンテ後のスナップショット」を参照する。
  - 例: `GraphSnapshot` (immutable) or `GraphRenderView` の導入。
- `restore_graph` は名称と責務を分割し、意図が明確な API とする。
  - 例: `apply_deletions()` / `reindex_vertices()` / `prune_edges()` など。

## 実施計画

### 1. 破壊的更新の分離方針を確定
- `restore_graph` の責務を分解して、明示的 API で呼べるように設計する。
- どのタイミングで `apply_deletions()` を呼ぶか決める。
  - 候補 A: `GraphEditorApp::update()` の最初に 1 回。
  - 候補 B: 破壊的操作が発生するイベント直後に呼ぶ。
- `encode()` が破壊的更新を行う必要があるかを検討し、必要なら事前に `apply_deletions()` を呼ぶ契約に変更する。

### 2. `restore_graph` の責務を分割・移動
- `Graph::restore_graph()` を以下のように再構成する。
  - `Graph::apply_deletions()`
    - `is_deleted` を持つ頂点・辺を削除する。
  - `Graph::reindex_vertices()`
    - 頂点IDの振り直しと辺のID補正。
- `restore_graph` を削除するか、内部で新 API を呼ぶラッパーにして段階的移行。
- 呼び出し元を更新し、描画関数から直接呼ばないようにする。
  - `src/components/central_panel.rs`

### 3. 描画はスナップショット参照のみに変更
- 描画に必要なデータ構造を「不変ビュー」として切り出す。
  - 例: `GraphSnapshot` (頂点/辺の clone + 参照は read-only)
  - 例: `Graph::snapshot()` を追加し、`draw_edges()` / `draw_vertices()` は snapshot 引数のみを受け取る。
- 既存の `draw_*` が `&mut Graph` を前提にしている箇所を整理する。
  - クリック/ドラッグなどは操作フェーズに残し、描画関数は描画のみ担当にする。
- `restore_graph()` の呼び出しが描画経路に存在しないことを確認する。

### 4. 明示的メンテナンス API の導入
- `apply_deletions()` を呼ぶ箇所を特定して配置する。
  - 例: `GraphEditorApp::update()` 内で描画前に 1 回。
  - 例: 削除モードのクリック処理完了後に呼ぶ。
- `encode()` / export 処理において、メンテナンスが必要なら呼び出し位置を整理。

### 5. 最小テスト/手動手順の用意
- 手動確認手順を用意 (docs/refactor/phase1_checklist.md 等でも可)。
  - 頂点削除 → エッジが削除されるか
  - 連続削除 → ID振り直し後に描画が崩れないか
  - `encode()` / export 実行で削除済み要素が含まれないか
  - 削除後にドラッグ/色変更/辺追加が正常に動くか
- 最低限の自動テスト案 (あれば):
  - `Graph::apply_deletions()` のユニットテスト (頂点/辺の整合性)

## 影響範囲と注意点
- `Graph` 内の `id` 再採番が描画や選択状態に影響するため、`EditMode::AddEdge` 等の状態更新のタイミングに注意。
- 描画ループから `restore_graph` を外すと、削除フラグ付き要素が一時的に残るため、描画で除外するか、スナップショット作成時に除外する必要がある。
- `export.rs` は `is_deleted` でフィルタしているため、破壊更新移動後も挙動が変わらないか確認が必要。

## 完了条件
- 描画関数は破壊的更新 API を呼ばず、スナップショット参照のみになっている。
- `apply_deletions()` が明示的なタイミングで呼ばれている。
- 最小の手動/テスト手順で Phase 1 の差分確認ができる。

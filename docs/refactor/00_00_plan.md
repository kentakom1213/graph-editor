# Refactoring notes

このドキュメントは、現状の設計上の課題と改善策を整理したものです。
主に保守性・拡張性・性能・テスト容易性の観点で調査しています。

## 課題と改善策

### 1. 状態が GraphEditorApp に集中している
- 課題: UI 状態、ドメイン状態、入出力、エクスポート状態が 1 つの struct に集約されている。
  - 影響: 変更影響範囲が広く、機能追加や修正が同一ファイルに集中しやすい。
- 改善: 状態を用途別に分割して責務を明確化する。
  - 例: `AppState` (ドメイン) / `UiState` (表示・入力) / `ExportState` (エクスポート中のイベント処理) のように分離。
  - `GraphEditorApp` はそれらを束ねるオーケストレーターに留める。

### 2. UI とドメインロジックが混在している
- 課題: `central_panel` で入力処理・グラフ操作・描画・幾何計算が同居している。
  - 影響: 描画ループ内でのロジックが肥大化し、バグ調査が難しい。
- 改善: 役割別にモジュール分割する。
  - 入力処理 (イベント解釈) / ドメイン操作 (グラフ変更) / 描画 (純粋レンダリング) を分離。
  - 例: `interaction/` (クリック/ドラッグ/選択) と `render/` を作成し、`GraphEditorApp` から呼ぶ。

### 3. Graph モデルに UI 状態が混在している
- 課題: `Vertex` / `Edge` に `is_pressed`, `is_selected`, `color`, `drag`, `z_index` など UI や表示に依存する情報がある。
  - 影響: ドメインとしてのグラフ表現が純粋でなく、操作やエクスポートに副作用が混入しやすい。
- 改善: ドメインモデルと表示モデルを分離する。
  - 例: `GraphModel` は純粋な頂点・辺のみ保持し、`GraphViewState` が選択や色などの UI 状態を保持する。

### 4. 描画処理がデータを破壊的に更新している
- 課題: 描画ループ内で `restore_graph` を呼び、削除フラグを持つ要素を実際に削除・再番号付けしている。
  - 影響: レンダリングが副作用を持ち、フレーム単位でデータ構造が変化する。デバッグが難しい。
- 改善: グラフのメンテナンス処理は明示的な操作に限定する。
  - 例: `apply_deletions()` を操作フェーズで呼ぶ。
  - 描画時は不変なスナップショットを参照する。

### 5. ID と参照解決が O(n^2) になりやすい
- 課題: 辺の描画時に頂点を `iter().find()` で探索しているため、描画フレームごとに O(n\*m) の探索が発生する。
  - 影響: 頂点数・辺数が増えると急激に描画が重くなる。
- 改善: 頂点参照の高速化と安定した ID を導入する。
  - 例: `Vec<Vertex>` の index を ID とせず、`HashMap<VertexId, Vertex>` を使う。
  - 辺は `VertexId` を保持し、描画時に O(1) で参照可能にする。

### 6. 変換行列の管理が分散している
- 課題: `Graph.affine` と `Vertex.affine` が共有されているが、適用のタイミングが散在しており理解しづらい。
  - 影響: 変換の二重適用や座標系の不整合が起こりやすい。
- 改善: 座標変換の責務を単一の場所に集約する。
  - 例: `TransformState` を導入し、全描画・入力座標を明示的に変換する。

### 7. エクスポート処理が UI に埋め込まれている
- 課題: エクスポートの状態管理、スクリーンショット処理、エラーハンドリングが `GraphEditorApp` 内にある。
  - 影響: UI と非同期処理が強く結合し、再利用やテストが難しい。
- 改善: エクスポートを独立したサービスに切り出す。
  - 例: `ExportService` を導入し、状態遷移とエラーを一元管理する。

### 8. 設定値とアルゴリズムの切り替えが拡張しづらい
- 課題: `AppConfig` が `Box<dyn Visualizer>` / `Box<dyn Simulator>` を保持しており、設定のシリアライズや UI 選択に不向き。
  - 影響: 設定保存や UI での切り替えが増えるほど実装負荷が高まる。
- 改善: 列挙型 + パラメータ構造体に置き換える。
  - 例: `VisualizerKind` + `VisualizerConfig` にして、UI で選択可能にする。

### 9. 重要ロジックのテスト基盤が不足している
- 課題: グラフ解析・配置・変換処理に対する単体テストが見当たらない。
  - 影響: リファクタリング時に挙動が壊れやすい。
- 改善: 重要なユースケースをカバーするテストを追加する。
  - 例: `BaseGraph::parse` の境界条件、`rebuild_from_basegraph` の頂点数・辺数検証など。

## 優先度の目安
1. 描画とデータ更新の分離 (課題 4)
2. モデル分離と状態分割 (課題 1, 3)
3. ID 解決と性能改善 (課題 5)
4. エクスポートと設定の整理 (課題 7, 8)
5. テスト追加 (課題 9)

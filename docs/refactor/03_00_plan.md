# Phase 3 Refactor Plan: ID 解決と性能改善

対象: docs/refactor/00_01_checklist.md の Phase 3 チェックリストを詳細化する。

## ゴール
- `VertexId` を導入し、頂点参照の安定性を確保する。
- 頂点参照を `HashMap<VertexId, Vertex>` などで O(1) 化する。
- `Edge` が `VertexId` を保持するように変更する。
- 描画時の頂点検索が O(n\*m) にならないことを確認する。
- 高速化のため，`fxhash::FxHashMap` を用いる

## 現状把握 (対象ファイル)
- `Vertex` は `id: usize` を保持し、`Vec<Vertex>` の index を前提としている。
  - `src/graph/structures.rs`
- 辺は `from/to: usize` の頂点IDを保持している。
- 描画やエクスポートで `iter().find()` による頂点探索が存在する。
  - `src/components/central_panel.rs`
  - `src/export.rs`

## 方針 (決めるべきこと)
- `VertexId` の型設計 (新規 struct or type alias)。
  - 例: `#[derive(Clone, Copy, Hash, Eq, PartialEq)] struct VertexId(usize)`
- `GraphModel` の内部表現を `HashMap<VertexId, Vertex>` ベースに変更する。
  - `Vec<Vertex>` を残す場合は別のインデックス構造を持つ。
- `Edge` は `VertexId` を保持し、描画/操作時に O(1) 参照できるようにする。

## 実施計画

### 1. `VertexId` の導入
- `VertexId` の型を決定し、新規型として定義する。
- 既存コードの `usize` ID を `VertexId` に置き換える方針を決める。
- `GraphViewState` など UI 側も `VertexId` を扱えるように設計する。

### 2. `GraphModel` のデータ構造変更
- `GraphModel` 内の `vertices` を `HashMap<VertexId, Vertex>` に変更する。
- `VertexId` の発番方法を定義する。
  - 例: `next_vertex_id` カウンタを保持
- 頂点削除時の挙動を整理 (ID再採番は行わない)。

### 3. `Edge` の参照変更
- `Edge` の `from/to` を `VertexId` に変更する。
- `GraphViewState` の edge view を `EdgeId` もしくは `(VertexId, VertexId)` ベースに変更する。

### 4. 描画・操作・エクスポートの更新
- 描画での `iter().find()` を廃止し、O(1) の参照に変更する。
  - 例: `HashMap<VertexId, Vertex>` から直接取得
- `GraphSnapshot` 生成時に `VertexId` ベースの参照に切り替える。
- `export.rs` も `VertexId` を前提にした描画ロジックへ移行する。

### 5. 動作確認
- 既存ユースケースで挙動が変わらないことを確認する。
  - 頂点追加/削除/辺追加/移動/色変更
  - エクスポート (PNG/SVG) で描画が崩れない
- `cargo check` を通す。

## 影響範囲と注意点
- `VertexId` 導入後は ID 再採番しないため、削除済み頂点の扱いを整理する必要がある。
- `GraphViewState` の同期処理が大きく変わる可能性がある。
- 描画順や `z_index` の扱いを維持するため、view 状態のマッピングを工夫する必要がある。

## 完了条件
- `VertexId` が導入され、`Edge` が `VertexId` を保持している。
- 描画/操作/エクスポートが O(1) 参照で動作する。
- `cargo check` が通る。

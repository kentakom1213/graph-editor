# 速度課題と対策メモ

本ドキュメントは Graph Editor の速度面での課題と、実装レベルの対策案を整理する。

## 主なボトルネック

### 1. 常時リペイントでアイドル時も描画が走る
- `src/update.rs` で `is_animated` が true の間 `ctx.request_repaint()` が毎フレーム呼ばれる。
- `is_animated` の初期値が true のため、何も操作していなくても CPU/GPU が回り続ける。

### 2. 物理シミュレーションの計算量が大きい
- `src/graph/simulator.rs` の反発力計算が各頂点に対して全頂点を走査し O(n^2)。
- 隣接頂点の列挙が `neighbor_vertices` で全エッジ走査 + 頂点探索を毎回行い O(n*m)。
- `clone()` を含む頂点コピーがループ内で発生するため不要なメモリ負荷がある。

### 3. フレーム毎のグラフ再構築・探索が多い
- `draw_edges` と `draw_vertices` の両方で `restore_graph()` が呼ばれ、毎フレーム削除処理と再番号付けが走る。
- `draw_edges` で毎フレーム `edge_count` を `HashMap` で数え直し、各エッジで頂点探索を行う。
- `draw_vertices` で `sorted_by_key` による並び替えが毎フレーム発生する。

### 4. 初期配置(可視化)の計算が重い
- `src/graph/visualizer.rs` の交差数評価が O(m^2)。
- `SimulatedAnnealing` では `max_iter=1000` 回の O(m^2) 評価が走る。
- 高密度用の回避はあるが、ノード数が多いケースでは依然重い。

### 5. エッジのヒットテストが重い
- `draw_edges` で全エッジに対して線/ベジェの距離計算を実施する。
  - → 曲線の辺はそれほど多くならないため，ベジェ曲線の距離計算は問題ない
- マウス移動時のみ必要な処理が毎フレーム行われる。

## 解決策の方向性

### 1. リペイント制御の改善
- `is_animated` をデフォルト false にする、またはアクティブ時だけ true にする。
- `ctx.request_repaint_after(Duration)` で一定間隔の再描画に抑える。
- 速度が十分小さいときに自動停止する（速度閾値で `is_animated` を false）。

### 2. 物理シミュレーションの高速化
- 隣接リストを事前構築し、`neighbor_vertices` の都度検索を避ける。
- `Vec<Pos2>` / `Vec<Vec2>` などの配列に位置・速度を分離してコピーをなくす。
- 反発力の計算を Barnes-Hut 等に置き換え、O(n log n) に近づける。
- 1フレームでの反復数を制限し、フレーム時間を上限管理する。

### 3. グラフ更新の差分化
- `restore_graph()` は削除/復元イベント時のみ実行し、フレーム毎には行わない。
- `edge_count` や `id -> index` のマップはグラフ変更時に再構築してキャッシュする。
- `z_index` の並びは必要時のみ更新し、描画時は安定ソートを避ける。

### 4. 初期配置の軽量化
- `max_iter` をノード数/エッジ数に応じて自動調整する。
- 交差数の厳密計算を近似（サンプリング）に切り替える。
- 大規模グラフは `Naive` 配置に切り替え、後から徐々に改善する。

### 5. ヒットテストの最適化
- マウスが動いた時のみエッジ距離計算を行う。
- 画面内のバウンディングボックスで候補エッジを絞る（空間インデックス）。
- ベジェの距離判定は近似直線に切り替えるなど、精度と速度のトレードオフを選べるようにする。

## 影響範囲のメモ
- 再描画制御: `src/update.rs`, `src/app.rs`
- 描画・ヒットテスト: `src/components/central_panel.rs`
- 物理シミュレーション: `src/graph/simulator.rs`
- 初期配置: `src/graph/visualizer.rs`
- グラフ構造と更新: `src/graph/structures.rs`
